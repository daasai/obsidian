# 1122平安问题复盘

摘要

### 事件时间线

- 平安银行系统升级时间为周五晚 21:00 至周六下午 14:00
- 周六中午 13:00，平安银行通知系统已恢复，但状态查询功能仍有延迟
- 14:00 时，团队发现交易状态仍显示"处理中"
- 15:21 时，发现约 560 笔交易需要处理
- 期间，客服接到多个用户紧急来电，陈鹤开始处理紧急交易
- 16:00-18:00，团队拉群进行沟通，组织人员进行交易状态和账户余额的人工核验
- 18:33，平安银行确认 560 笔交易均未出金，团队决定重新发起交易
- 19:22，所有出金处理完毕

### 问题分析

- 对银行核心系统升级的重视度不足，事前未制定专门预案
- 银行通知的系统状态未自动更新，导致团队依赖错误信息
- 未在交易入口设置拦截，用户仍能发起交易请求
- 处理时缺乏人员值守，导致问题发现后才临时召集人手
- 耗时较长，从问题发现到解决花费约 6 小时
- 缺乏及时的客户沟通机制和统一话术

### 改进措施

- 对银行系统维护采取"不可信"原则，提前做好预案
- 银行系统维护期间主动暂停相关业务功能
- 系统恢复后先进行小流量测试，确认无问题后再完全开放
- 完善客户通知机制，在C端增加维护提醒
- 提前收集官方信息并提供给客服，统一对外回应
- 可见期内（至少一季度或半年）执行最谨慎原则处理所有相关事务

### 行动项目

- [ ]  建立银行系统维护预案，包括提前暂停业务和分步恢复机制
- [ ]  优化异常处理流程，加强内外部沟通机制
- [ ]  在客户端界面增加系统维护提醒功能
- [ ]  完善银行维护通知的接收和监控机制
- [ ]  制定客服统一话术和官方信息发布流程
